Bastion · Zero-Trust Principles · MFA

Threat Model & Mitigations

The lab is designed around realistic threat scenarios. Each threat category below maps to one or more concrete mitigations already implemented in the environment.

Compromised IoT Device
e.g. vulnerable firmware, default credentials
  • VLAN 20 has no path to LAN or MGMT — lateral movement blocked at L3
  • pfBlockerNG blocks C2 domain resolution before TCP connection forms
  • Suricata detects anomalous outbound traffic patterns from IoT net
  • Wazuh alerts on any cross-VLAN traffic that bypasses rules
External Brute Force / Scanning
e.g. SSH scans, VPN password spray
  • WAN default-deny — only UDP 1194 (OpenVPN) exposed externally
  • VPN requires client certificate; password alone is insufficient
  • Suricata detects port scans and known scanner signatures
  • Wazuh brute force rule triggers Slack alert on auth failure threshold
  • GeoIP pfBlockerNG restricts inbound by country-of-origin
Compromised LAN Workstation
e.g. phishing, drive-by download
  • LAN → MGMT is default-deny, with one narrow, documented exception scoped to a single host and port — not a general path
  • Bastion is the only MGMT ingress; reaching it from any other LAN device would require a second, separate compromise
  • pfBlockerNG blocks malware domain resolution on LAN
  • Suricata detects C2 beaconing and exploit kit traffic on LAN egress
  • Wazuh FIM detects if critical files are modified
Credential Theft / Reuse
e.g. stolen password used to access services
  • LDAP (AD DS on the Hyper-V host) centralizes credentials — one place to disable a compromised account
  • VPN requires certificate + credentials; password alone is not enough
  • All admin access to pfSense and Wazuh is logged and audited
DNS-Based Attacks
e.g. DNS tunneling, malware C2 via DNS, typosquatting
  • pfBlockerNG DNSBL sinkholes malicious domains at resolver level
  • DNS-over-TLS to upstream resolver prevents interception/manipulation
  • Wazuh rule detects abnormally long DNS query labels (tunneling signal)
  • Suricata DNS anomaly rules catch protocol-level tunneling patterns
Unauthorized Physical / VLAN Access
e.g. rogue device plugged into switch, VLAN hopping
  • Managed switch enforces 802.1Q VLAN boundaries
  • MGMT uses static DHCP assignments
  • Unused ports are in the disabled state

Design Decisions

Why isolate infrastructure inside a dedicated Management VLAN?
Administrative systems represent the highest-value assets in the environment. All management interfaces, pfSense, Wazuh, and bastion host reside within a dedicated management network that is inaccessible from user and IoT segments by default. Access is restricted through a bastion host, creating a separate administrative security boundary that significantly reduces opportunities for lateral movement following workstation compromise.

The one deliberate exception is a single firewall rule allowing a specific Rustdesk session on LAN to reach the Bastion's admin port — nothing else on LAN can. Pure isolation isn't operationally realistic for a one-person lab; the goal was to make the exception as narrow as possible and document why it exists, rather than pretend it doesn't.

Security Principles: Defense-in-Depth, Least Privilege, Administrative Segmentation
Why implement default-deny VLAN segmentation?
The network is designed using zero-trust principles where trust is never inherited based solely on network location. LAN, IoT, and Management segments are isolated using VLANs and restrictive firewall policies. All inter-VLAN communication requires explicit authorization, preventing compromised devices from freely traversing the environment and limiting the blast radius of a security incident.

Security Principles: Zero Trust, Network Segmentation, Least Functionality
Why use certificate-based VPN authentication?
Remote access represents one of the most common attack vectors for both enterprise and home environments. OpenVPN requires both user credentials and client certificates, ensuring that password compromise alone cannot provide network access. Combined with limited subnet exposure and centralized authentication, this approach significantly reduces the effectiveness of brute-force attacks, password spraying, and credential reuse.

Security Principles: Strong Authentication, Identity-Centric Security
Why centralize monitoring through Wazuh?
Detection and response capabilities are built around centralized visibility. Security telemetry from Windows systems, Linux hosts, pfSense, Suricata, VPN services, and network infrastructure is aggregated into Wazuh for correlation, alerting, and investigation. This enables faster detection of malicious activity while providing MITRE ATT&CK mapping, audit trails, and incident response workflows comparable to enterprise SOC operations.

Security Principles: Continuous Monitoring, Detection Engineering, Defense-in-Depth
Why enforce DNS filtering and resolver control?
DNS is a frequent command-and-control channel used by malware and adversaries. All client DNS traffic is redirected to pfSense and processed through pfBlockerNG, preventing devices from bypassing security controls through external resolvers. DNSBL sinkholing, threat intelligence feeds, and encrypted upstream DNS significantly reduce exposure to malicious infrastructure and DNS-based command-and-control activity.

Security Principles: Secure Name Resolution, Preventive Controls
Why forward Wazuh alerts to Slack instead of relying on the dashboard alone?
A SIEM nobody is actively watching is just an log archive. Severity filtered Slack alerts mean critical and high-severity detections — auth failure thresholds, cross-VLAN violations, FIM changes — surface immediately instead of waiting to be noticed on the next dashboard review. Filtering by severity at the webhook, rather than forwarding everything, keeps the channel meaningful instead of becoming background noise that gets tuned out — which is its own detection-evasion risk.

Security Principles: Continuous Monitoring, Reducing Dwell Time, Alert Fidelity
Why self-host Rustdesk instead of using a commercial remote-access tool?
Commercial remote-access tools route session traffic through a third-party relay by default, which means trusting that vendor's infrastructure, uptime, and security posture. Self-hosting the relay keeps all session data on-prem and removes a dependency that's outside this network's control — a meaningful consideration given how often remote-access software itself becomes the attack vector rather than the thing being defended.

Security Principles: Reducing Third-Party Trust, Data Locality, Supply Chain Risk Reduction
Why disable unused switch ports and use static DHCP reservations on MGMT?
Both are small, unglamorous controls that close a real gap: an open switch port is an invitation for a rogue device, and dynamic DHCP on a sensitive segment means any device that gets physical access also gets an IP. Disabling unused ports and reserving addresses by MAC on MGMT means a device has to be known and provisioned before it can talk on that network at all.

Security Principles: Physical Security, Least Functionality, Asset Visibility
